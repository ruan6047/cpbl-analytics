| 痛點 | 對應 |

## A. 全站 UI/UX 原則（八條，驗收時逐條對照）

1. **快速理解優先（數據可視化的核心）**：問題不在看不清楚，在**資料過多造成判讀負荷**。
   （ruan6047 07-11：「重點是讓使用者能快速理解數據；過多數據導致沒辦法快速理解」）
   - **預設只給結論層，細節漸進揭露**：表格預設核心欄位（其餘展開/切換）；區塊預設摘要、點開才是全量。
   - **數字不裸列**：一律帶判讀輔助——PR 條/色階/趨勢箭頭/聯盟均值對照（站內已有 `PercentileBar`/`prColor`，擴大使用）。孤立數字無法快速判讀。
   - **每個區塊回答一個問題**：配 eyebrow 小標＋（必要時）一句話結論；答不出「這區塊讓使用者知道什麼」就砍或併。
   - **5 秒測試**＝頁面卡首要驗收：打開頁面 5 秒內能答出該頁主問題（誰領先？今天誰贏？這選手行不行？）。
   - **標準頁面解剖**：h1＋一行副標 → 主角區（每頁只有一個主角）→ 輔助區塊依重要性排序，禁無標題裸區塊。
   - 易讀性下限（次要但仍驗）：數據主體 ≥14px、對比 ≥4.5:1、Outfit + `tabular-nums`；圖表必有軸標/圖例/單位。
2. **三層底色，不新增灰階**：`paper`（頁底）→ `surface`（卡）→ `surface-2`（卡內強調區）。
   分界一律 `border-line`。任何新頁面不得引入第四層灰或自定色票。
3. **顏色必有語意，禁止裝飾性用色**：`accent`＝行動/焦點、**隊色＝身分**（chip/光暈/圖表線）、
   `up`/`down`＝數據好壞、amber＝狀態警示（延賽/保留）。同一語意全站同色。
4. **誠實 UX**（承專案紅線）：預測必附基準對照（如全押主場）；資料缺口顯式標注
   （TrackMan 場地覆蓋、逐打席 2018+）；不做假精確（該 `—` 就 `—`，不補 0）。
5. **一致的元件語彙**：同一概念全站同一長相——eyebrow 小標、dl 堆疊網格（決勝資訊式）、
   隊色 chip、狀態徽章、`.card`。**新頁面先找既有元件，禁止平行發明**。
6. **行動端一等公民**：375px 無橫向溢出＝鐵則（頁級 `scrollWidth ≤ viewport`）；
   寬表格二選一（sticky 首欄 或 卡片化），觸控目標 ≥44px；grid 必鎖 `grid-cols-1` 基底。
7. **動效節制而有目的**：只在入場與狀態變化動（標準 0.3s；圖表入場 0.8s）；
   全部受 `prefers-reduced-motion` 約束；不做無限循環裝飾動畫。
8. **感知效能**：skeleton/empty/error 三態統一元件；圖表 lazy；切換不得佈局塌陷（CLS）。

**驗收方法標準**（每張子卡通用）：量測而非目視——computed style、boundingRect、
scrollWidth、console 零錯誤；行動端 375px + 桌機 1280 雙檔；`tsc` 綠。

## B. 拆卡（順序＝依賴序：通用 → 個別；大 → 小）

### 通用層（先做，個別頁面全吃它的產出）

| 卡 | 範圍 | 驗收重點 |
|---|---|---|
| **UX-2 設計 tokens 與原則落地＋深色模式** | 原則 A 寫進 `globals.css` 註解＋盤點修正：硬編色清查（UI-1 稽核曾發現 **19 檔硬色**）、spacing/字級規範化、色彩語意對照表；**深色模式併入此卡**（ruan6047 07-11 裁定：tokens 層一次到位，含 `data-theme` 切換與暗色 token 組） | 全站 grep 無 hex 硬色（隊色/圖表語意色除外）；深淺兩色系逐頁截圖驗收；tokens 有註解文件 |
| **UX-3 共用元件標準化** | `ui.tsx` 收斂：eyebrow 小標、dl 網格、chip、狀態徽章、skeleton/empty/error 三態、**DataTable**（寬表容器+sticky 首欄+表頭一致封裝） | 以下「模組化基準」數字顯著改善且至少 3 頁換裝無回歸；元件有 props 文件 |
| **UX-4 骨架與導覽＋頁面解剖規範** | header 導覽資訊架構（10 項導覽的分組/優先序）、**標準頁面解剖落地**（h1+副標/主角區/輔助區塊排序——直接回應「區塊混亂」）、footer、行動端導覽 | 375px 導覽可用性；各頁標題與區塊結構一致 |

#### 模組化基準（07-11 審計；UX-3 驗收對照這組數字）

| 指標 | 現況 | 說明／目標 |
|---|---|---|
| 卡片殼 `rounded-xl border border-line` inline | **×46** | `Card` 元件存在但 46 處繞過手寫 → 換裝後 <10（特例註記） |
| 手寫 `<table>` | **×22，散佈 9 檔** | 零表格元件；sticky 首欄（UI-4）只套了戰績表 → 全數走 `DataTable` |
| `overflow-x-auto` 手寫 | ×18 | 溢出保護靠自律（LIVE-1 曾踩 grid 溢出）→ 併入 DataTable/寬容器元件 |
| 載入/空態 ad-hoc | 10 檔各寫各的，skeleton＝0 | → 三態元件統一 |
| components vs app 行數 | 1,926 vs 5,016（28%）| 原子件 14 個但無組織級元件（表格/區塊/三態）→ 頁面檔膨脹（最大 515 行） |

> 結論：**「頁面不統一」的機制性根因就是模組化不足**——樣式改一次要動 46+ 處，必然漂移。
> UX-3 是本案槓桿點；先抽元件，頁面卡（UX-5〜9）才是「換裝」而非重寫。

## C. 流程約定

- 每張子卡獨立走 `分支 → 自測 → 獨立查核 → merge`；查核者 ≠ 執行者。
- 全部 ⚪ 一般卡（無統計/ML 紅線），視覺驗收同家族異 session 可；建議關鍵頁人審。
- 部署節奏照 memory `no-deploy-during-layout-tweaks`：頁面層子卡累積滿意再一次上線，
  不逐卡部署（通用層 UX-2/3/4 因影響面大，各自完成即部署驗證一次）。
